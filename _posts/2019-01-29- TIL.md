# 백지노트

## 널포인트 세이프

### `?` 키워드를 사용한 Null 허용 타입 만들기

```java
val s1:String? = "Hello, Kotlin" // ?를 자료형 뒤에 붙임으로써 null이 가능한 변수로 만들어진다. 당연히 널이 아닌 값이 들어와도 이상무
val s2:String? = null // 널이 들어가도 컴파일 시 NullPointerException이 발생하지 않는다. 
val s3:String = null // 컴파일 에러 발생. 당연하지
val s4:String = s2 // 컴파일 에러 발생. s4는 널이 가능하지 않기 때문에 s2가 들어갈 수 없다. s2는 널이 들어올 수도 있으니까? Type missmatched
```

`?` 를 이용해서 널이 가능한 변수를 만들어 준다. 이를 활용하면 컴파일 시에 널을 잡고 갈 수 있기 때문에 런타임 오류를 줄일 수 있다.


### `?.` 연산자 활용

```java
if(a != null) println(a.length) //a가 널이 아니면 a의 길이를 출력한다.
```
나는 자바에서 위와 같이 매번 `null`인지 아닌지 확인해 가면서 코드를 작성했었다. 하지만 코틀린에서는 이러한 과정을 획기적으로 줄여줄 수 있다. `?.` 연산자를 활용하면 다음과 같이 간단하게 작성할 수 있다.

```java
println(a?.length) //a가 널이 아니라면 length를 참조하여 길이를 반환한다. 만약 a가 널이라면 그냥 null값을 반환. 뒤의 함수를 참조하지 않게 된다. null일 때 다른 함수를 참조하려고 하면 NullPointerException을 만나게 될테니...
```

연속으로 사용할 수도 있다.

```java
println(a?.substring(5)?.length) // a가 null이 아니라면 substring을 참조하고 그 결과가 null이 아니라면 length를 참조한다. 중간에 null이 발생하면 그냥 null값을 반환한다.
```

만약 자바였다면 위의 코드가 이런식이었을 것이다.

```java
if (a != null){
	val b:String? = a.substring(5)
	if(b != null)
		println(b.length)
}
```

코드의 길이 뿐만 아니라 코드의 가독성 역시 코틀린이 훨씬 낫다. 쭉 읽으면서 바로바로 의미가 해석되는데 심지어 길이까지 훨씬 짧다. 코틀린 짱

### `?:` 엘비스 연산자

엘비스 연산자는 `?:` 를 사용한다. 엘비스 연산자 좌항의 값이 `null`이 아니라면 좌항의 값이 반환되고 `null`이라면 우항의 값이 반환된다.
이 엘비스 연산자 때문에 생활코딩 페이스북에 질문을 했었는데 재미있는 경험이었다.

```java
fun main(args: Array<String>) {
    //의도: s1이 null이기 때문에 엘비스 연산자에 의해서 ?:의 우항이 반환됨
    val s1:String? = null
    println(s1?.length ?: 0)
    //결과: 0

    //의도: s2가 null이 아니기 때문에 ?:연산자의 좌항인 "Hello, Kotlin"의 길이가 반환됨
    val s2:String? = "Hello, Kotlin"
    println(s2?.length ?: s2=="I love Java")
    //결과: false
    //질문: 왜 엘비스 연산자의 좌항결과가 아닌 우항의 결과가 반환될까요?
}
```

엘비스 연산자는 결국 연산자이기 때문에 우선순위와 괄호(?)를 잘 이해해고 사용해야 한다. 위의 코드에서 문제가 되었던 것은 엘비스 연산자의 좌항이 `null`이 아니기 때문에 그냥 좌항의 값이 반환될 것이라고 생각했는데 자꾸 s2=="I love Java"의 값이 (우항) 반환되는 것이었다. 생코 전문가 분(?)께서 한참 고민을 하시다가 내려주신 결론은 앞에서부터 차례대로 해석해야 정답이 나온다는 것이었다. 나도, 그 분도, 심지어 그 분의 회사 사람들도 엘비스 연산자의 좌항 우항에만 포커스를 맞추다 보니 좌항 우항을 제대로 구분하지 못했던 것이다. 지금 생각해보면 너무나 당연하고 간단한 문제였는데 그 때는 왜 그렇게 보이지 않던지...

그나마 다행인 것은 나의 고민(?)이 그 분들께도 어느정도 혼란을 드렸다는 점... ㅋㅋㅋ

그리고 뭔가 인간미가 느껴져서 좋았다. 이런 작은 실수를 전문가들도 하는구나... 아직 실력은 미천하지만 언젠가는 나도 비슷한 자리까지 갈 수 있겠다는 확신이 들었던 에피소드 였다.
